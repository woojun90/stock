# ETF轮动策略 - 技术设计文档

## Context

### 背景

当前InStock系统已具备完整的股票/ETF数据获取、技术指标计算、交易执行等基础设施。本项目在此基础上构建AI Agent驱动的ETF动量轮动策略，实现：

- 基于动量效应的ETF轮动信号生成
- 多维度风险控制机制
- AI增强的市场环境感知与决策解释

### 现有系统约束

| 约束 | 说明 |
|-----|------|
| 数据源 | 东方财富网，通过 `eastmoney_fetcher` 封装 |
| 指标计算 | TA-Lib + pandas，32种技术指标 |
| 交易执行 | easytrader，支持多券商 |
| 数据存储 | MySQL，已有ETF数据表 |
| 运行环境 | Python 3.11，支持Docker |

### 利益相关方

- 策略使用者：需要清晰的投资信号和风险提示
- 系统运维：需要稳定的定时任务和告警机制
- 开发者：需要可扩展的模块化架构

---

## Goals / Non-Goals

### Goals

1. **实现6个核心Skills**
   - 每个Skill独立可测试、可复用
   - 清晰的输入输出接口
   - 支持AI Agent调用

2. **构建规则配置系统**
   - YAML格式配置文件
   - 热更新支持（无需重启）
   - 参数化策略逻辑

3. **集成现有交易框架**
   - 继承 `StrategyTemplate`
   - 复用 `MainEngine` 和 `EventEngine`
   - 最小化代码侵入

4. **支持回测验证**
   - 核心逻辑确定性可回测
   - AI增强部分仅实盘启用
   - 回测结果可重现

### Non-Goals

1. **不开发新的数据源** - 复用现有东方财富数据
2. **不修改现有策略** - 与海龟交易等策略并行运行
3. **不实现全自动交易** - AI建议需人工确认关键决策
4. **不支持高频交易** - 周频调仓，日频监控

## Decisions

### Decision 1: Skills模块独立于策略模块

**决定**: Skills放置在 `instock/skills/` 目录，独立于 `instock/core/strategy/`

**理由**:
- Skills是功能性模块，可被多个策略复用
- 策略模块是业务逻辑，调用Skills组合能力
- 解耦后Skills可独立测试、独立演进

**替代方案**:
- ❌ 放在 `core/strategy/` 下 - 策略和技能概念混淆
- ❌ 放在 `trade/` 下 - Skills不限于交易，也用于分析

```
instock/
├── skills/           # 独立的Skills模块
│   ├── __init__.py
│   ├── etf_data.py
│   ├── momentum.py
│   └── ...
├── core/
│   └── strategy/
│       └── etf_rotation.py  # 调用skills
└── trade/
    └── strategies/
        └── etf_rotation_strategy.py  # 继承StrategyTemplate
```

---

### Decision 2: 规则配置使用YAML格式

**决定**: 策略规则使用YAML配置文件，运行时加载

**理由**:
- 人类可读，便于调整参数
- 支持嵌套结构，表达复杂规则
- Python原生支持，无需额外依赖
- AI Agent可直接读取和理解规则

**替代方案**:
- ❌ Python代码配置 - 修改需重启，不灵活
- ❌ JSON格式 - 不支持注释，可读性差
- ❌ 数据库存储 - 查询开销，不便于版本控制

```yaml
# config/etf_rotation_rules.yaml
signal_rules:
  momentum_period: 21
  thresholds:
    strong_signal: 1.0
    weak_signal: 0
```

---

### Decision 3: 回测与实盘统一代码路径

**决定**: 回测和实盘使用相同的Skills代码，通过模式参数区分

**理由**:
- 避免回测代码与实盘代码不一致
- 单一代码路径减少维护成本
- 回测验证的逻辑就是实盘执行的逻辑

**实现方式**:
```python
# Skills统一接口
def run(mode='backtest'):  # mode: 'backtest' | 'live'
    signal = momentum_skill.calculate(data)
    
    if mode == 'live':
        # 仅实盘运行AI增强
        context = market_context_skill.analyze()
        signal = enhance_signal(signal, context)
    
    return signal
```

**AI参与边界**:
| 功能 | 回测模式 | 实盘模式 |
|-----|---------|---------|
| 动量计算 | ✅ 执行 | ✅ 执行 |
| 趋势过滤 | ✅ 执行 | ✅ 执行 |
| 风控检查 | ✅ 执行 | ✅ 执行 |
| 市场环境感知 | ❌ 跳过 | ✅ 执行 |
| 决策解释 | ❌ 跳过 | ✅ 执行 |
| 人工干预建议 | ❌ 跳过 | ✅ 执行 |

---

### Decision 4: 数据库表设计遵循现有模式

**决定**: 新增3张表，命名和结构遵循 `tablestructure.py` 现有风格

**理由**:
- 保持代码一致性
- 复用现有的数据库操作封装
- Web展示模块可自动识别新表

**表结构**:
```python
# etf_rotation_signal - 信号记录
# etf_rotation_position - 持仓记录
# etf_risk_event - 风险事件
```

---

### Decision 5: AI Agent通过Skill接口调用

**决定**: AI Agent不直接操作数据库或交易接口，而是通过Skills封装的能力执行

**理由**:
- Skills提供明确的输入输出契约
- AI只需理解Skill的用途和参数
- 便于权限控制和日志记录
- Skills可独立测试验证

**调用流程**:
```
AI Agent
    │
    ▼
┌─────────────────┐
│ Skill Interface │
│ def execute()   │
└─────────────────┘
    │
    ├──▶ etf_data_skill.get_realtime()
    ├──▶ momentum_skill.calculate()
    ├──▶ risk_skill.check()
    └──▶ rebalance_skill.execute()
```

## Risks / Trade-offs

### Risk 1: AI Agent决策不可重现

**风险**: AI推理结果不确定，无法在回测中复现实盘决策

**缓解措施**:
- 核心策略逻辑（动量、趋势、风控）使用确定性代码
- AI仅参与增强决策（市场环境、决策解释），不改变核心信号
- 记录AI决策过程日志，支持事后审计

### Risk 2: 规则配置热更新导致状态不一致

**风险**: 运行中修改配置，部分模块使用旧配置，部分使用新配置

**缓解措施**:
- 配置变更时触发全局重新加载
- 关键配置变更记录日志并通知
- 提供配置版本回滚机制

### Risk 3: Skills依赖循环

**风险**: 多个Skills相互调用导致循环依赖

**缓解措施**:
- 定义清晰的Skills依赖层次（数据层 → 计算层 → 决策层 → 执行层）
- 禁止跨层反向调用
- 代码审查时检查依赖关系

### Risk 4: 交易执行延迟

**风险**: 14:54触发调仓，执行时已接近收盘，可能影响成交

**缓解措施**:
- 提前计算信号，14:30开始准备
- 使用限价单而非市价单
- 记录滑点，优化执行时机

### Trade-off: 灵活性 vs 复杂性

**权衡**: 规则配置化增加了灵活性，但也增加了系统复杂度

**选择**: 优先保证灵活性，通过良好的文档和默认配置降低使用门槛

1. 设计YAML配置结构
2. 实现配置加载与热更新
3. 验证配置正确性

1. 开发 `etf_rotation.py` 策略核心
2. 集成到交易框架
3. 回测验证

### 阶段4: 实盘部署

1. 配置券商接口
2. 小资金测试
3. 监控与告警配置
4. 正式运行

### 回滚策略

- 保留旧策略文件，可通过配置切换
- 数据库变更使用增量表，不影响现有数据
- 提供一键禁用新策略的配置项
